# Risk Register — CTFGenerator Productization

Milestone 0 deliverable. Initial risk register seeded from the productization plan's
known issues and the codebase map. Rows describe **current** behavior grounded in the
codebase map; mitigations reference the **planned** milestone(s) (`M<n>`) that own the
fix. "Target" language marks aspirational state.

**Legend**
- **Likelihood / Impact**: Low / Medium / High.
- **Category**: security · reliability · operational · product · schema.
- **Current status**: Open (unaddressed) · Partial (some mitigation exists) · Accepted (known, deferred).
- **Owning milestone**: where the fix lands per the release stages. Plane names refer to the
  four target planes (Author Studio, Competition Control Plane, Execution Plane, Evaluation Lab).

## Risk table

| Risk ID | Category | Description | Likelihood | Impact | Current status | Mitigation / owning milestone |
|---|---|---|---|---|---|---|
| R-01 | security | Generated, vulnerable-by-construction challenge code executes **in the same process/host as the control surface**. `runtime_validator._run` shells out to `docker compose` and runs bundle-shipped `private/solver.py` / `tests/healthcheck.py` **on the host with caller privileges by default** (`--sandbox` is opt-in). `replay_validator`, `sibling_validator`, `scenario_runtime`, `agent_eval` each reach into `runtime_validator`'s private helpers. | High | High | Open | **CRITICAL.** Split into isolated Execution Plane; control plane never executes challenge code, never mounts Docker socket. Worker protocol + job system (M7/M8). Interim: `validate-runtime --sandbox`. |
| R-02 | reliability | **No persistence layer / storage abstraction.** State is scattered: JSONL append log (`events.py`, bare `threading.Lock`), optional `postgres_events.py`, per-bundle `variant.json`/report files read ad hoc. `serve` defaults to `InMemoryEventStore` unless `--events-file` given → all competition state lost on restart. | High | High | Open | PostgreSQL domain model + repository abstraction + migrations; scoreboards reconstructable from persisted score events (M6). |
| R-03 | schema | **No real schema versioning.** Versioning is three independent hard-coded `"1.0"` constants (`SPEC_VERSION` in `models.py`, `SCHEMA_VERSION` in `report_writer.py`, `__version__`) with no shared registry, no negotiation, no migration, no upgrade path. `spec.json` (`spec_to_dict`) and `variant.json` carry **no** version field at all; **no consumer reads or validates any version constant** — they are write-only stamps. | Medium | High | Open | Real schema versioning: versioned spec/artifact envelopes, compatibility checks, migration path (M4). |
| R-04 | security | **Renderer path-escape surface.** MCP write tools resolve `output_dir` via `_resolve_in_workspace` against a workspace root (good), but the CLI `create`/`create-from-spec` paths and `force=True` (`shutil.rmtree` first) provide an arbitrary host write + recursive-delete primitive outside the MCP sandbox. Key invariant "generated paths cannot escape build dir" is only enforced on the MCP seam today. | Medium | High | **Mitigated (M3)** | DONE: `build.py` enforces path validation (no `..`/absolute/control/confusable/reserved), managed-dir-only recursive delete (`.ctfgen-build` marker), dangerous-root refusal, atomic move-aside publish, and size/count limits at **all** generation entry points (adversarially reviewed). Residual: cross-team/host isolation at runtime still needs the Execution Plane (M7/M8). |
| R-05 | security | **Flag reachable via image layer / build inputs.** Flag flows into compose via `${CTFGEN_FLAG:-}` at runtime (good), but `.env.example` ships `CTFGEN_FLAG=...` sample and the vuln class is baked into service source; risk of a real flag being committed to an image layer or artifact rather than injected only at launch. | Medium | High | Partial | Immutable content-addressed published artifacts; flag injected only at instance launch; artifact scanning; ZERO public flag leakage invariant (M3 hardening / Execution Plane M7). |
| R-06 | security | **Public scoreboard token passed in URL query string.** `dashboard_server` uses `parse_qsl`; `serve --public-token` (or an auto-generated token printed once) authenticates the public scoreboard/feed. Tokens in query strings leak via logs, proxies, referrers. | Medium | Medium | Open | Replace with header/cookie-based public access under the production API; structured logging that never logs tokens (M8+ Control Plane API). |
| R-07 | security | **Hand-rolled auth/session in `dashboard_server.py`** (705-line bespoke `ThreadingHTTPServer` with its own session login, token rotation, `secrets`, cookie handling). `--secure-cookie` only meaningful behind a TLS-terminating proxy; the built-in server is plain HTTP. No framework-provided authn/authz, CSRF, or rate limiting. | Medium | High | Open | Replace with maintained ASGI framework (FastAPI or comparable) + real authz for the 8 user roles; reverse proxy + TLS (M8+ Control Plane; UI M12+). |
| R-08 | operational | **PEP 668 externally-managed host cannot `pip install` deps.** Optional deps (`mcp`, `anthropic`/`openai`, `psycopg`) are lazy-imported and raise install-hint errors; on a PEP 668 host the documented `pip install ctf-generator[...]` remedy fails without a venv, blocking MCP/LLM/Postgres paths. | Medium | Medium | Partial | Documented venv/container install path; packaged release artifacts + supported deployment model (M1 release artifacts; V1 containerized deploy). |
| R-09 | reliability | **Single-node execution, no concurrency/isolation model.** No worker pool, no job queue, no leases/heartbeats/retries. Execution is blocking `subprocess` + `urllib` loops smeared across five modules; cannot meet 25 concurrent teams / 20 active challenges / ≥99% launch success. | High | High | Open | PostgreSQL-backed job queue (`FOR UPDATE SKIP LOCKED`, leases, heartbeats, retries, idempotency, dead-letter); isolated worker hosts (M7/M8). |
| R-10 | product | **"AI-resistance" metric currently overclaims.** `score.py` reports an "AI-resistance score /100" from static bundle heuristics (`variant_uniqueness`, `statefulness`, `solver_depth`, `live_interaction`, `scanner_resistance`) — a static proxy, not a measured agent outcome. The CLI `score` band ("strong"/"good"/…) reads as a validated resistance guarantee. | Medium | Medium | Open | Rename/reframe as a heuristic quality signal; separate measured agent evaluation in the Evaluation Lab (M19 rename). |
| R-11 | security | **MCP ↔ CLI security boundary is convention-enforced, not structural.** `mcp_server` deliberately never imports `scenario_runtime`/`agent_eval`/`dashboard_server`/`subprocess` and exposes only pure tools; but this is guaranteed by import discipline, not architecture. A future edit could silently widen the model-host attack surface. | Low | High | Partial | Make the safe-tool boundary a real architectural seam (application services + explicit adapter layer); keep Docker/exec off the MCP surface (M3/M7 refactor). |
| R-12 | reliability | **Deterministic-rebuild guarantee is unverified in CI.** Key invariant: identical (generator version, spec, family version, seed) ⇒ identical artifacts. `meta_mapping()` avoids wall-clock, but there is no gate proving byte-identical rebuilds; families carry no independent version. | Medium | High | **Mostly closed (M0/M3)** | DONE: golden baseline fixtures + `test_baseline_fixtures.py` rebuild every family in-process and assert byte-identical output (required CI gate); all 8 families confirmed deterministic. Residual: per-**family** versioning (independent of generator version) lands in M4. |
| R-13 | operational | **Fragile config parsing: write-only YAML, no reader.** `yaml_writer.py` emits YAML but there is no reader; `validator.py`/`families.py` re-parse `challenge.yaml` with indentation-sensitive regex (`family_of`, `_SCENARIO_LINE`, marker checks). Minor formatting drift silently breaks family resolution and validation. | Medium | Medium | Open | Adopt a real YAML library + typed spec parsing in the domain layer (M3/M4 refactor). |
| R-14 | operational | **`cli.py` is a 1389-line god-module** that imports nearly every subsystem and inlines command logic — no separation between arg parsing and use-case orchestration. Raises regression risk and blocks CLI/API/MCP sharing one code path. | Medium | Medium | Open | Extract thin application-layer use-cases; `cli`/`api`/`web`/`mcp` become thin adapters; CLI and API share application services (M3 refactor / M8 API). |
| R-15 | security | **Untrusted bundle code runs even under `--sandbox` with limited guarantees.** `--sandbox` runs healthcheck/solver in an ephemeral read-only `python:3.11-slim` container, but the default path does not, and network isolation / resource enforcement / expiration / cleanup are not systematically enforced by the current runtime path. | Medium | High | Partial | Rootless Docker/Podman + rootless BuildKit on isolated workers; network isolation, resource limits, instance lifecycle + reconciliation (M7). |
| R-16 | reliability | **No cross-team instance isolation model.** No concept of per-team instances, ownership, or access control in the current runtime/dashboard path. Invariant "one team cannot access another team's instance" is unmet. | Medium | High | Open | Instance orchestration keyed to (team, challenge, competition) in Control + Execution planes (M7/M8). |
| R-17 | security | **Solve-uniqueness invariant unenforced.** Invariant: a correct submission creates AT MOST one solve per (team, challenge, competition). `events.py`/`scoreboard.py` compute over appended events but nothing enforces single-solve idempotency at submission time. | Medium | Medium | Open | Idempotent submission handling + DB constraint in the persistent domain model (M6/M8). |
| R-18 | operational | **No audit trail for privileged state changes.** Invariant: every privileged state change auditable; admin score changes require explicit reason. Current `dashboard_server` has no structured audit log; score/config edits are not reason-gated or persisted as auditable events. | Medium | Medium | Open | Append-only audit log + reason-gated admin actions in Control Plane (M8/M12). |
| R-19 | security | **Secrets/keys risk in logs and reports.** Invariant: flags/session-tokens/provider-keys NEVER logged. Reports (`report_writer`) serialize solver logs and runtime output verbatim (`serialize_runtime` → `logs`); LLM backends take provider keys; no redaction layer exists. | Medium | High | Partial | Structured JSON logging with redaction; scrub flags/tokens/keys from reports and logs (M1 hardening / M8). |
| R-20 | product | **`live_adversarial_engine` knob advertised but unwired.** `AIResistance.live_adversarial_engine` defaults `False` and is documented as a "Phase-5 knob, unwired"; challenge.yaml surfaces it. Risks organizers expecting live adversarial defense that does not run. | Low | Medium | Accepted | Clarify status in docs; wire (or scope out per V1 non-goal "autonomous unrestricted AI defense") in Evaluation Lab work (M19+). |
| R-21 | reliability | **No backup/recovery story.** No defined RPO/RTO; in-memory-by-default event store and scattered file state make the RPO 5min / RTO 30min targets unattainable today. | Medium | High | Open | PostgreSQL persistence + backup/restore + recovery drill (M6; validated at v1.0). |
| R-22 | operational | **Private grader artifacts co-located with player bundle.** `private/` (flag, `solver.py`, `variant.json`, `solution.md`, scenario timeline) ships inside the same generated bundle as `public/` content. Trust boundary is directory-convention only; a mis-served bundle leaks solvers/flags. Invariant "private solvers never served to contestants" is enforced only by convention. | Medium | High | Partial | Content-addressed public/private artifact separation; publication step strips private files before contestant delivery (M3 artifacts / M8 publication). |
